计算机体系结构学习笔记 — 第5章 线程级并行

原著: Computer Architecture: A Quantitative Approach, 6th Edition — John L. Hennessy & David A. Patterson

线程级并行（TLP）是多核和多处理器系统的核心。本章覆盖多核架构、缓存一致性、内存一致性模型、同步和并行编程模型。

5.1 引言 — 为什么需要 TLP

从 ILP 到 TLP 的转变

2003 年之后，ILP 的收益递减导致行业转向 TLP：

时期 主要并行策略 代表处理器
1990s ILP（单核超标量） Pentium Pro, P-II
2000s 芯片级多处理 (CMP) Core 2 Duo, Opteron
2010s 众核 (Many-core) Xeon Phi, GPGPU
2020s 异构多核 Apple M1, Snapdragon

核心驱动因素：

  1. Dennard scaling 终结 → 无法继续提高频率
  2. ILP 收益递减 → 更宽的发射宽度回报很低
  3. 更多晶体管可用来集成多个核
  4. 并行应用（服务器、ML 训练）需要更多的核

TLP 的基本分类

分类 描述 例子
SMP 对称多处理，多个相同核共享内存 多核 CPU
CMP 芯片级多处理，同一芯片上多个核 Intel Core, AMD Ryzen
SMT 同时多线程，一个核执行多线程 Hyper-Threading
NUMA 非均匀内存访问 多插槽服务器
MPP 大规模并行处理 超级计算机集群
🧓
多核设计的本质是：与其做一个功耗 100W 的复杂大核，不如做 4 个功耗 25W 的简单核。你之前做的 ARM 芯片，从双核到八核的演进就是这个逻辑。 做中端综合时，多核的设计会让时钟分布和复位网络复杂一个数量级——每个核要单独做 PPA 优化，再加上核间互联的一致性协议逻辑。

5.2 多核架构基础

共享内存 vs 分布式内存

特性 共享内存 分布式消息传递
通信方式 共享变量 显式消息
硬件需求 缓存一致性 网络接口
编程难度 相对简单 困难（显式通信）
可扩展性 有限（~100 核） 好（数千核）
典型实现 多核 CPU 集群、MPP

多核芯片的互连

总线 (Bus)：

  • 所有核共享总线
  • 简单但带宽有限
  • 适合 2–4 核

交叉开关 (Crossbar)：

  • 全连接，点对点
  • 面积随核数 O(n²) 增长
  • 适合 4–8 核

环形 (Ring)：

  • 核间通过环网连接
  • 延迟随跳数增加
  • Intel 常用（~20 核）

网格 (Mesh)：

  • 2D 网格连接
  • Tile 架构，可扩展
  • 延迟随曼哈顿距离增加
  • 众核（>64 核）常用

私有缓存 vs 共享缓存

缓存层级 典型架构 优点 缺点
L1 私有（每核） 低延迟 一致性开销
L2 私有或共享 折中 -
L3 共享（LLC） 减少跨核数据复制 延迟高（多核访问）

LLC (Last-Level Cache) 划分策略：

  • 静态划分：每核固定大小
  • 动态划分：根据负载动态调整（Intel CAT 技术）

5.3 缓存一致性协议

为什么需要一致性

当多个核同时缓存同一内存地址时：

核 1 写 x = 1    → L1[核1] x=1 (Modified)
                   L1[核2] x=0 (Stale!)
核 2 读 x         → 得到 x=0 (错误！)

一致性 (Coherence) 保证：每个核看到的共享变量在任何时刻都是一致的。

MESI 协议详解

四个状态：

状态 含义 其他核有副本？ 与主存一致？
M Modified 脏数据，仅本核持有 否 否
E Exclusive 干净数据，仅本核持有 否 是
S Shared 干净数据，多个核持有 是 是
I Invalid 数据无效 - -

状态转换（核心事件）：

         PrRd (本地读)
    ┌─────────────────────────┐
    │     I ←─────────── S   │  PrRd/其他核共享
    │      ↕                  │
    │ PrWr (本地写)          │  PrWr/总线读独占
    │      ↕                  │
    │     M ←─── E ←─── S    │
    │                        │
    BusRdX (其他核请求写)    │  BusRd (其他核请求读)
    │      ↕                  │
    └─────────────────────────┘

监听协议 (Snooping)

每个缓存控制器监听所有核的访存请求：

写失效协议 (Write-Invalidate)：写一个缓存行时，失效所有其他核中的副本。

核 1 写 x = 1:
  1. 发送总线写失效 (BusRdX)
  2. 所有其他核失效其 L1 中的 x
  3. 核 1 的 L1 中 x → Modified

核 2 读 x:
  1. 发送总线读 (BusRd)
  2. 核 1 的缓存监听到请求
  3. 核 1 写回主存，状态 → Shared
  4. 核 1 和核 2 的 L1 中 x → Shared

目录协议 (Directory-based)

不适合大规模系统的原因：监听总线的广播开销随核数 O(n²) 增长。

目录协议使用集中式/分布式目录跟踪每个缓存行的状态：

  • 每个内存块的目录项记录哪些核缓存了该块
  • 写请求只通知有该块的核（非广播）
  • 消息量 O(核数) 而不是 O(核数²)
🧓
实际做多核芯片设计时，一致性协议是验证最困难的部分之一。MESI 已经有 4 个状态，但 MOESI 有 5 个状态，每一种状态转换都有数十个 corner case。你做过形式验证的话应该明白——缓存一致性协议的 FV 是最经典的应用场景之一。

MOESI 协议

在 MESI 基础上增加了 O (Owned) 状态：

状态 含义
O Owned 数据脏，但其他核可以有共享副本

优点：Modified 的数据被其他核读时，不需要写回主存，直接从拥有者（Owner 核）转交。

一致性协议的优化

技术 工作原理 效果
写缓冲 延迟写操作，批量发送 减少一致性消息
写合并 合并多个同一行的写请求 减少总线流量
预测嗅探 推测其他核可能请求的块并提前响应 减少延迟
监听过滤器 过滤不需要响应的总线事务 减少功耗

5.4 内存一致性模型

一致性 vs 正确性

缓存一致性保证同一地址的写最终对所有核可见。 内存一致性模型定义多个不同地址的读写操作在什么顺序下对程序员可见。

顺序一致性 (SC)

最简单的模型，也是最严格的：

所有核的访存操作按程序顺序执行，所有核看到相同的全局顺序。

核 1:   核 2:
A = 1   B = 1

// 合法结果：(A=1, B=1)
// 非法结果：(A=0, B=0) // 不可能两个核都没看到对方的值

SC 的代价：严格限制重排序，性能损失大。现代处理器几乎不使用 SC。

宽松一致性 (Relaxed Consistency)

允许在某些条件下重排序访存操作，使用内存屏障 (Memory Fence/Barrier) 强制顺序。

模型 允许重排序 性能 编程难度
TSO (Total Store Order) 允许读→写重排序 中等 适中的
PC (Processor Consistency) 允许更多重排序 较高 较难
RC (Release Consistency) acquire/release 语义 最高 最难

x86 使用 TSO：允许后续的读在之前的写完成之前执行。 ARM/Power 使用 RC：更放松，性能更高，但需要程序员或编译器插入更多的 barrier。

1
2
3
4
5
6

// RC 模型下的同步示例
lock:
    acquire(mutex);   // 确保后续的访存不会被重排序到 acquire 之前
    // 临界区
    x = x + 1;
    release(mutex);  // 确保前面的访存在 release 之前完成

内存屏障语义

屏障类型 作用
DMB (Data Memory Barrier) 保证屏障前后的所有访存按序执行
DSB (Data Synchronization Barrier) 等待所有访存完成
ISB (Instruction Synchronization Barrier) 清空流水线，保证指令缓存一致性
Store Barrier 保证所有 store 在屏障前的操作完成
Load Barrier 保证所有 load 在屏障前的操作完成

5.5 同步 (Synchronization)

基本概念

并行编程中的三种同步需求：

  1. 互斥 (Mutual Exclusion)：一次只有一个线程访问临界区
  2. 条件同步 (Condition Synchronization)：等待某个条件成立
  3. 内存同步 (Memory Synchronization)：保证共享变量在正确的时间点可见

锁的实现

Test-and-Set (TAS)：

int test_and_set(int *lock) {
    int old = *lock;
    *lock = 1;
    return old;
}

void lock(int *lock) {
    while (test_and_set(lock))
        ; // 自旋等待
}

void unlock(int *lock) {
    *lock = 0;
}

问题：每个 TAS 操作都会写缓存行 → 大量一致性消息。

Test-and-Test-and-Set (TTAS)：

1
2
3
4
5
6
7

void lock(int *lock) {
    while (1) {
        while (*lock == 1);  // 先读（共享状态）
        if (!test_and_set(lock)) // 再写
            break;
    }
}

TTAS 显著减少了总线流量：读操作在共享状态下进行，不产生一致性消息。

Ticket Lock：

typedef struct {
    int ticket;
    int serving;
} ticket_lock;

void lock(ticket_lock *l) {
    int my_ticket = atomic_fetch_add(&l->ticket, 1);
    while (l->serving != my_ticket);
}

void unlock(ticket_lock *l) {
    atomic_fetch_add(&l->serving, 1);
}

公平性：保证 FIFO 顺序，不会出现线程饥饿。

MCS Lock：

自旋锁的问题：每个等待线程都在不停轮询。MCS 锁使用链表结构，每个节点在自己的本地变量上自旋，减少了一致性流量。

原子操作

硬件的原子指令是实现锁的基础：

指令 效果
LL/SC (Load-Linked/Store-Conditional) 对同一地址读后写，中间有其他写入则条件写失败
CAS (Compare-And-Swap) 比较并交换
FAA (Fetch-And-Add) 原子加法
SWP (Swap) 原子交换

RISC-V 使用 LR/SC 和 AMO (Atomic Memory Operation)：

1
2
3
4
5
6

// RISC-V LR/SC 实现的原子加法
again:
    lr.w t0, (a0)       // 加载链接
    add  t0, t0, a1     // 运算
    sc.w t1, t0, (a0)   // 条件存储
    bnez t1, again      // 失败则重试

无锁数据结构

不使用锁，只用原子操作保证并发安全。

  • 无锁队列 (Lock-Free Queue)：用 CAS 操作入队/出队
  • 免等待 (Wait-Free)：每个操作在有限步内完成，不会被其他线程阻塞
  • RCU (Read-Copy-Update)：读操作不需要锁，更新时复制后原子替换指针

5.6 同时多线程 (SMT, Hyper-Threading)

SMT 的工作原理

一个物理核内部运行多个线程，共享核内资源（功能单元、缓存），但每个线程有自己的寄存器状态。

时间 →
标准超标量: IF | ID | EX| WB | （单一指令流）
SMT (2线程): T0:T1:T0:T1:...（交错指令流）

SMT 的优点

  1. 利用核内资源的空闲期（如缓存缺失时的等待时间）
  2. 提高核的利用率（典型 15–30% 性能提升）
  3. 硬件开销小（主要是寄存器文件翻倍）

SMT 的成本

  • 面积增加 ~5%（寄存器 + 追踪状态）
  • 单线程性能可能下降 ~5–10%（资源竞争）
  • 安全漏洞（Hyper-Threading 的侧信道攻击）

粗粒度多线程 vs 细粒度多线程 vs SMT

类型 切换时机 资源利用率
粗粒度 长延迟事件（如缓存缺失） 中等（切换需要清空/填满流水线）
细粒度 每周期切换 高（但单线程进度慢）
SMT 交错发射，不显式切换 最高（指令级交错）

并行编程的常见问题

问题 描述 解决方案
数据竞争 多个线程不加同步地读写同一变量 锁、原子操作
死锁 线程 A 等 B 的资源，B 等 A 的资源 锁排序、超时
活锁 线程在不断重试但无人能前进 退避策略
饥饿 某线程永远无法获取资源 公平锁
假共享 不同核的线程修改变量在同一缓存行 缓存行对齐

假共享 (False Sharing) 详解 

1
2
3
4
5
6

// 假共享：a 和 b 可能在同一缓存行
int a[NUM_THREADS], b[NUM_THREADS];

// 线程 0 写 a[0]
// 线程 1 写 a[1]
// → 两个在不同核上频繁的写导致缓存行在两个核的 L1 之间反复 bouncing

2

// 缓存行对齐（64B）
int __attribute__((aligned(64))) a[NUM_THREADS];

5.8 综合案例：多核处理器的性能分析

Amdahl 定律在多核上的应用

Speedup=1(1−f)+f/NSpeedup = \frac{1}{(1 - f) + f / N}Speedup=(1−f)+f/N1​

其中 f 是可并行部分的比例，N 是核数。

Gustafson 定律（更现实）：

Speedup=N−f×(N−1)Speedup = N - f \times (N - 1)Speedup=N−f×(N−1)

当问题规模随核数增加而增大时，并行效率不会受 Amdahl 定律限制太久。

可扩展性

强扩展 (Strong Scaling)：固定问题规模，增加核数 → 受 Amdahl 定律限制

弱扩展 (Weak Scaling)：问题规模随核数线性增加 → 每核负载不变

真实世界的数据

应用 并行效率 (16核) 瓶颈
稠密矩阵乘法 95%+ 内存带宽
稀疏线性代数 50–80% 内存延迟（不规则访问）
图分析 20–60% 通信开销、负载不均衡
MapReduce 70–90% 中间数据 shuffle

5.9 常见谬误与陷阱

  1. 认为核数翻倍性能翻倍：Amdahl 定律 + 一致性开销 + 内存带宽限制
  2. 忽略假共享的影响：即使是完全无关的变量，在同一缓存行也会导致严重性能下降
  3. 假设所有内存访问延迟相同：NUMA 架构中访问远端内存延迟高 50–100%
  4. 误认为 SMT 总是有利的：高负载场景下 SMT 可能因为资源竞争而降低总吞吐
  5. 低估锁竞争的开销：高并发锁竞争会导致严重的性能下降（锁退化）
  6. 忽视缓存一致性消息对功耗的影响：大量一致性消息在芯片高速总线上消耗可观功耗
  7. 假设顺序一致性模型：大多数处理器使用宽松一致性模型，需要显式的内存屏障

5.10 总结

本章核心要点：

  1. TLP 是后 Dennard 时代性能增长的主要驱动力，多核设计成为通用处理器的主流
  2. 缓存在私有和共享之间做 trade-off，LLC 是缓存一致性的关键设计点
  3. MESI/MOESI 协议保证多核缓存一致性，监听协议（小规模）和目录协议（大规模）是两种实现方式
  4. 内存一致性模型（TSO/RC）定义了访存操作的全局可见性顺序
  5. 同步机制（锁、原子操作、Barrier）是实现互斥和条件同步的硬件基础
  6. SMT在一个核内执行多线程，以少量硬件开销提高核的利用率
  7. 并行编程从 OpenMP（共享内存）到 MPI（消息传递）到 CUDA（GPU），覆盖从多核到众核到异构
