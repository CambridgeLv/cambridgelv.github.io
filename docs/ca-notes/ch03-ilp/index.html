计算机体系结构学习笔记 — 第3章 指令级并行

原著: Computer Architecture: A Quantitative Approach, 6th Edition — John L. Hennessy & David A. Patterson

指令级并行（ILP）是超标量处理器的核心。本章覆盖流水线、分支预测、乱序执行、寄存器重命名、多发射等技术。

3.1 引言 — 什么是指令级并行

ILP 的基本概念

指令级并行 (Instruction-Level Parallelism) 是指程序中指令之间可以重叠执行的程度。两条指令如果互相独立（没有数据依赖、控制依赖或资源冲突），就可以并行执行。

ILP 的来源：

  • 流水线 (Pipelining)：在不同阶段重叠执行多条指令
  • 多发射 (Multiple Issue)：每个周期发射多条指令
  • 乱序执行 (Out-of-Order Execution)：不按程序顺序执行，提高利用率
  • 猜测执行 (Speculation)：在不确定结果时先执行
🧓
ILP 是你做 RV32I 核心最核心的部分。流水线、转发、分支预测，这些在附录 C 里有基础概念，第三章则把这些提升到高级技术层面——超标量、乱序、VLIW。 简单来说：如果附录 C 教你做 5 级流水线，那第三章教你怎么做那些能跑 SPEC 的商用处理器。

流水线回顾

5 级 RISC 流水线（附录 C 基础）：

IF → ID → EX → MEM → WB

流水线冒险 (Hazards)：

  • 结构冒险：硬件资源冲突
  • 数据冒险：指令间数据依赖
  • 控制冒险：分支指令导致的取指不确定性

数据冒险分类 (Tomasulo 标记法)

依赖类型 名称 含义 转发方案
RAW 真依赖 (Read After Write) 后一条读取前一条的结果 必须转发/等待
WAR 反依赖 (Write After Read) 后一条覆盖前一条要读的寄存器 寄存器重命名消除
WAW 输出依赖 (Write After Write) 两条写同一寄存器 寄存器重命名消除

3.2 流水线的高级技术

深流水线

现代处理器采用 14–20+ 级流水线（甚至更深），核心原因：

  • 每级逻辑更少 → 频率更高
  • 但分支误预测惩罚更大
  • 转发逻辑更复杂
  • 流水线停顿开销更大

示例：Intel Pentium 4 (NetBurst) 有 20–31 级流水线，频率达到 3.8GHz，但因分支误预测惩罚太大，总体 IPC 并不理想。

多发射 (Multiple Issue)

每个周期发射多条指令。两种实现方式：

特性 静态多发射 (VLIW) 动态多发射 (超标量)
指令分组 编译器完成 硬件完成
数据依赖检查 编译器 硬件（乱序窗口）
硬件复杂度 低 高
向后兼容 差（指令格式改变） 好
典型代表 Intel Itanium x86, ARM, RISC-V 超标量

超标量处理器的基本结构：

取指 → 译码 → 重命名 → 发射 → 执行 → 写回 → 提交
       ↓                              ↓
    (并行译码n条)              (多个功能单元并行执行)

VLIW (Very Long Instruction Word)

VLIW 将多条独立操作打包成一条长指令，由编译器确保指令间的独立性。

优点：

  • 硬件简单（不需要复杂的调度逻辑）
  • 功耗效率高
  • 可预测性强

缺点：

  • 编译器压力大
  • 代码膨胀（空操作填充）
  • 向后不兼容（指令格式绑定深度）
🧓
VLIW 在 DSP 和 GPU 中很常见，通用处理器已经几乎放弃了（Itanium 是前车之鉴）。但最近 AI 加速器的指令调度又有点 VLIW 的风格——编译器提前排好管线。这说明 VLIW 在领域专用场景下其实很有效。

3.3 分支预测

分支预测的重要性

对于 15 级流水线的处理器：

  • 每 5–6 条指令就有一条分支
  • 分支方向的确定一般在 EX 级完成
  • 每次分支误预测浪费 10+ 个周期
  • 误预测率每降低 1%，性能提升约 2–3%

静态分支预测

策略 预测率 描述
总是跳转 50–60% 向后分支（循环）预测跳转
总是不跳转 40–50% 简单实现
向后跳转/向前不跳转 ~70% 常见循环分支准确
配置文件引导 ~80% 用 profile 数据指导编译期预测

动态分支预测

1-位饱和计数器

记录上次分支的结果，预测相同方向。准确率 ~80%（对循环的最后一次预测失败）。

2-位饱和计数器

有 4 个状态：强不跳转、弱不跳转、弱跳转、强跳转。需要连续两次误预测才翻转。

    ┌─────────┐    不跳转    ┌─────────┐
    │强不跳转 │ ←───────── │弱不跳转 │
    └────┬────┘            └────┬────┘
         │                      │
    跳转 │                 跳转 │
         │                      │
         ▼                      ▼
    ┌─────────┐    不跳转    ┌─────────┐
    │弱跳转   │ ←───────── │强跳转   │
    └─────────┘             └─────────┘

相关预测器 (Correlating Predictor)

利用最近分支的历史来预测当前分支。因为分支结果往往相关：

1
2

if (x > 0) { ... }   // 分支 b1
if (y > 0) { ... }   // 分支 b2 — 结果常与 b1 相关

$(m,n)$ 预测器：使用最近 m 个分支的全局历史作为索引，每个历史模式对应一个 n-bit 饱和计数器。

  • (1,2)：1 位历史 + 每历史 2 位计数器 = 2 个 2-bit 表
  • (2,2)：2 位历史 + 每历史 2 位计数器 = 4 个 2-bit 表

Tournament 预测器

组合全局预测器和局部预测器，再有一个选择器在两者之间选择。

Alpha 21264 的 Tournament 预测器：

  • 局部预测器：基于每条分支的单独历史
  • 全局预测器：基于全局分支历史
  • 选择器：使用 2 位计数器选择预测器

总预测准确率可达 95% 以上。

混合预测器 (TAGE)

TAGE (TAgged GEometric) 是现代处理器的主流预测器：

  • 使用多个不同长度的全局历史表
  • 长的历史表捕捉长时间的相关性
  • 短的捕捉刚到期的相关性
  • 每个表带 tag（标签位）避免别名

TAGE 在各种 SPEC 程序中都能达到 96%+ 的预测率。

3.4 乱序执行与寄存器重命名

为什么需要乱序执行

当一条指令因为数据依赖而停顿，后续无关指令应该可以跳过它继续执行。

乱序执行的核心思想：尽可能快地执行就绪的指令，而不是必须按程序顺序。

Tomasulo 算法

Tomasulo 算法的三个关键组件：

  1. 保留站 (Reservation Stations)：等待就绪的操作数，负责监视公共数据总线 (CDB)
  2. 公共数据总线 (Common Data Bus)：广播结果给所有等待它的保留站
  3. 寄存器重命名 (Register Renaming)：通过寄存器别名表 (RAT) 消除 WAR 和 WAW 依赖

操作流程：

1. 指令发射：从指令队列取指令
   → 检查是否存在功能单元的保留站空位
   → 从 RAT 读取寄存器 tag（不是值！）
   → 如果操作数就绪（在寄存器中），直接拿到值
   → 如果不就绪（在某个保留站中等待），记录该保留站的 tag

2. 指令执行：当保留站中的操作数全部就绪后
   → 执行运算
   → 结果放到 CDB 上

3. 写结果：通过 CDB 广播结果
   → 所有等待该结果的保留站拿到值
   → RAT 更新
   → 提交逻辑记录
🧓

Tomasulo 是硬件调度器的经典。你在 RV32I 核心里暂时用不到这个（那是乱序处理器的范畴），但理解它的思想很重要：

  • 寄存器重命名把架构寄存器和物理寄存器解耦，让硬件能自由调度
  • 保留站类似于你写 RTL 时的 issue queue/FIFO
  • CDB 的广播设计在高频率下是个时序挑战——如果一个 cycle 内太多结果要广播，fanout 太大

寄存器重命名的实现

架构寄存器 (Architectural Register)：程序员/编译器看到的寄存器（如 x0–x31）

物理寄存器 (Physical Register)：硬件实际的寄存器文件（可能 64–256 个）

RAT (Register Alias Table)：将架构寄存器映射到物理寄存器

原始程序：          after renaming：
add x1, x2, x3  →  add p40, p38, p39
add x1, x4, x5  →  add p41, p42, p43  // x1 → 新的物理寄存器
add x6, x1, x7  →  add p44, p41, p45  // 使用最新的映射
  • 第二条指令对 x1 的写不会因为 WAR/WAW 依赖而阻塞第一条指令
  • 第三条指令的 x1 自动指向最新的版本（p41）
  • 第一条指令的结果写入 p40 而不是 p41，互不干扰

重排序缓冲 (ROB)

ROB (Reorder Buffer) 是乱序处理器的提交点：

  • 指令按程序顺序写入 ROB
  • 乱序执行结束后通过 CDB 写入 ROB
  • 只有最旧的指令的物理寄存器结果才能提交到架构寄存器
  • 支持精确异常（exception 发生时，ROB 中所有后续指令结果被丢弃）

精确异常 (Precise Exception)

乱序处理器必须保证异常发生时，处理器的状态与程序顺序执行到异常点一致。

ROB 使精确异常成为可能：

  1. 异常发生时，标记引发异常的 ROB 条目
  2. 清空 ROB 中所有更新指令（不提交到架构状态）
  3. 将架构状态恢复到异常点之前
  4. 跳转到异常处理程序

3.5 猜测执行 (Speculation)

控制猜测

分支预测 + 猜测执行：预测分支方向，沿预测路径取指和执行。

恢复机制：

  1. 分支误预测时，清空预测路径上所有指令的 ROB 条目
  2. 分支指令后的所有架构寄存器恢复
  3. 从正确目标地址重新取指

数据猜测

  • 加载猜测 (Load Speculation)：假设加载不会触发异常
  • 值预测 (Value Prediction)：预测指令的结果值（预测正确则跳过执行）
  • 内存依赖预测 (Memory Dependency Prediction)：预测加载和存储之间的顺序关系

猜测执行的问题

  1. 功耗浪费：误预测路径上的所有指令消耗了功耗但没产生有效结果
  2. 安全漏洞：Spectre / Meltdown 攻击利用的是猜测执行的副作用对缓存的影响
  3. 硬件复杂度：恢复机制需要足够的缓冲和清空逻辑
🧓
Spectre/Meltdown 这两个漏洞就是猜测执行的直接后果——CPU 在预测路径上执行的指令可能把不应该有权限的数据加载到缓存中，攻击者通过时间差就能推断出缓存状态。 做架构设计时，猜测执行的安全性已经成为一个必须考虑的因素，而不只是性能和功耗。

3.6 超标量实现技术

发射宽度与 IPC

处理器 发射宽度 典型 IPC 发布年份
Intel Pentium 2 0.5–1.0 1993
Intel Core 2 4 1.5–2.5 2006
Intel Skylake 6 2.0–3.5 2015
Apple M1 (Firestorm) 8 3.0–4.5 2020

IPC 瓶颈：

  • 指令级并行有限（平均 2–4 条无关指令/窗口）
  • 分支预测不够完美
  • 缓存缺失增加
  • 功能单元不够

取指与译码

  • 取指带宽：每周期取 16–32 字节指令（x86 变长指令需要复杂的前端解码）
  • 指令缓冲：取指队列缓存预取的指令
  • 预解码：在译码前标记指令边界和分支

功能单元组成

功能单元类型 典型数量 延迟
整数 ALU 2–6 1 cycle
整数乘法 1–2 3–5 cycles
分支单元 1–2 1–2 cycles
浮点加 1–2 3–4 cycles
浮点乘 1 4–5 cycles
加载/存储 2–3 3–4 cycles (L1 hit)

调度窗口 (Issue Window / Scheduler)

处理器 调度窗口大小
Intel Core 2 ~32 entries
Intel Skylake ~97 entries
Apple M1 ~630 entries

更大的调度窗口 → 更多的待调度的就绪指令 → 更高的 ILP 利用 → 更大的功耗和面积

全局调度 (Global Scheduling)

跨越基本块的指令调度，需要处理控制流。

追踪调度 (Trace Scheduling)：编译器选择一条最热门路径（trace）作为假设的执行路径，沿该路径全局调度指令，对偏离路径的情况插入补偿代码。VLIW 编译器常用。

超级块 (Superblock) 与超块 (Hyperblock)

  • 超级块：只有一个入口（优化后的追踪路径），多个出口
  • 超块：通过 if-conversion 将条件分支转换为谓词代码后的基本块集合

3.8 功耗感知的 ILP 设计

ILP 的设计在功耗方面代价很大：

功能 功耗占比（典型处理器）
取指/译码 15–25%
调度/乱序窗口 15–25%
寄存器文件 5–15%
分支预测 2–5%
功能单元 10–20%
缓存/存储层次 25–40%

ILP 的收益递减：

  • 从 2 发射到 4 发射：IPC 提升 ~60%，功耗增加 ~100%
  • 从 4 发射到 8 发射：IPC 提升 ~30%，功耗增加 ~200%

这就是为什么现代处理器在追求 ILP 的同时必须考虑每瓦性能。

3.9 综合案例：Apple M1 的 ILP 设计

Apple M1 (Firestorm 大核) 是 ILP 设计的现代典范：

  • 发射宽度：8（解码宽度 8）
  • 调度窗口：~630 entries（远大于竞争对手）
  • ROB：~630 entries
  • 物理寄存器：~384
  • 功能单元：6 ALU, 3 分支, 3 加载, 2 存储
  • L1 指令缓存：192KB (！)
  • 分支预测：TAGE 衍生的混合预测器

M1 Firestorm 的 IPC 在 SPEC06 中达到 3.5–4.5，远高于同期的 x86 处理器（2.0–3.0），说明大调度窗口 + 高发射宽度 + 好的分支预测 = 高 IPC。

3.10 常见谬误与陷阱

  1. 假设 IPC 与发射宽度成正比：实际 IPC 受限于程序内的 ILP（平均不到 4），盲目增加发射宽度没用
  2. 忽视分支预测的全局影响：每 1% 的误预测率损失可能对应 1–2% 的性能
  3. 认为深流水线总是好的：Pentium 4 的经验证明，深度需要平衡
  4. 忽略功耗墙：ILP 具有功耗收益递减特征
  5. 假设乱序执行总能提高性能：媒体和 DSP 处理中，序执行配合编译器调度可能更高效
  6. 低估寄存器重命名的面积代价：物理寄存器文件和 RAT 的功耗可观
  7. 忽视存储层次对 ILP 的影响：内存延迟决定了指令调度窗口至少需要覆盖缓存缺失惩罚
  8. 认为编译器可以替代硬件调度：静态调度和动态调度各有优势，两者互补

3.11 总结

本章核心要点：

  1. ILP 是通过重叠执行独立指令来提高性能，其来源包括流水线、多发射、乱序执行和猜测执行
  2. 数据依赖（RAW/WAR/WAW）是限制 ILP 的根本因素，寄存器重命名消除 WAR 和 WAW
  3. 分支预测是提升 ILP 的关键——深流水线和高发射宽度使预测准确率的重要性倍增
  4. 乱序执行通过 Tomasulo 算法的保留站 + CDB + RAT 实现，ROB 支持精确异常
  5. 超标量宽度和调度窗口大小决定硬件能利用的 ILP 上限
  6. 编译器优化（循环展开、软件流水、全局调度）补充硬件的 ILP 挖掘能力
  7. ILP 有收益递减效应— 功耗和复杂度随发射宽度超线性增长，IPC 提升亚线性增长
