计算机体系结构学习笔记 — 附录C 流水线：基础与中级概念

本文覆盖 5 级 RISC 流水线的全部基础与中级概念，与我们的 RV32I 核心项目直接对应。

C.1 流水线入门

什么是流水线

流水线 (Pipelining) 允许多条指令的执行在时间上重叠。每个步骤称为一个流水级 (pipe stage)。

5 级 RISC 流水线：

流水级 功能 缩写
IF 取指令 (Instruction Fetch) IF
ID 指令译码 + 读寄存器 (Instruction Decode) ID
EX 执行 / ALU 运算 (Execute) EX
MEM 数据内存访问 (Memory Access) MEM
WB 写回寄存器 (Write Back) WB

流水线的性能方程

无流水线：

Latency=N×Cycles_per_instruction×Cycle_TimeLatency = N \times Cycles\_per\_instruction \times Cycle\_TimeLatency=N×Cycles_per_instruction×Cycle_Time

有流水线（理想情况）：

Throughput=N+Stages−1Stages≈N    (when  N≫Stages)Throughput = \frac{N + Stages - 1}{Stages} \approx N \;\; (when\;N \gg Stages)Throughput=StagesN+Stages−1​≈N(whenN≫Stages)

理想情况下，每周期完成一条指令，吞吐量提升 5 倍。

流水线冒险 (Hazards)

冒险是阻碍流水线连续运行的三种情况：

  1. 结构冒险 (Structural Hazard)：硬件资源冲突
  2. 数据冒险 (Data Hazard)：指令依赖关系
  3. 控制冒险 (Control Hazard)：分支/跳转导致的不确定性

结构冒险

当多条指令同时需要同一个硬件资源时发生。

示例：统一指令和数据缓存的处理器中，Load 指令（MEM 级）和下一条指令的取指（IF 级）都要访问同一块 SRAM。

解决方案：哈佛架构（分离的指令 + 数据 L1 缓存），或插入气泡（Stall）。

C.2 数据冒险与转发

三类数据依赖

// RAW (Read After Write) — 真依赖
ADD x1, x2, x3    // 写 x1
SUB x4, x1, x5    // 读 x1 ← 必须等待 ADD 写完

// WAR (Write After Read) — 反依赖（寄存器重命名可消除）
SUB x4, x1, x5    // 读 x1
ADD x1, x2, x3    // 写 x1 ← 读之后才写，没问题

// WAW (Write After Write) — 输出依赖
ADD x1, x2, x3    // 写 x1
SUB x1, x4, x5    // 写 x1 ← 最终结果是 SUB 的

流水线中只有 RAW 是真依赖，需要硬件处理。WAR 和 WAW 在 5 级流水线中不会出现（WB 是最后一个级，不可能重排）。

流水线中的 RAW 冒险

考虑 5 级流水线：

1
2
3

ADD x1, x2, x3    // IF ID EX MEM WB
SUB x4, x1, x5    //    IF ID ── EX MEM WB
                        ↑ 需要 x1，但 x1 还在 EX 级

不使用转发时：需要在 ID 级检测到依赖后插入 2 个气泡（stall）：

1
2
3

ADD x1, x2, x3    // IF ID EX MEM WB
SUB x4, x1, x5    //    IF ID ST ST EX MEM WB
                                        ↑ 3 周期后才拿到 x1

转发 (Forwarding / Bypassing)

转发是将 ALU/内存的计算结果直接从当前流水级传递给需要它的流水级，而不是等写回寄存器后再读。

     ID→EX   EX→MEM   MEM→WB
        ↓        ↓        ↓
   ┌────┴────────┴────────┴──────┐
   │      转发逻辑 (MUX)         │
   └────────┬────────────────────┘
            ↓
          EX 阶段的 ALU 输入

转发路径：

转发的源 转发的目标 延迟节约
EX 结果 → EX ALU 入口 下一条 ALU 指令 2 cycles
MEM 结果 → EX ALU 入口 隔一条的 ALU 指令 1 cycle
MEM 结果 → MEM 访存地址 Load 后紧跟着的 Store 1 cycle
WB 结果 → EX ALU 入口 隔两条的 ALU 指令 0 (已写回)

转发检测逻辑

转发控制器的核心是比较寄存器的地址：

// 转发检测：EX 转发到 EX
if (EX/MEM.RegisterRd != 0 
    && EX/MEM.RegisterRd == ID/EX.RegisterRs1) 
    forwardA = 10; // 从 EX/MEM 转发

// MEM 转发到 EX
if (MEM/WB.RegisterRd != 0 
    && MEM/WB.RegisterRd == ID/EX.RegisterRs1) 
    forwardA = 01; // 从 MEM/WB 转发

五种转发情况：

情况 源指令位置 目标指令位置 转发路径
1 EX EX EX→EX
2 EX→MEM EX MEM→EX
3 MEM→WB EX WB→EX
4 EX MEM (Store) EX→MEM
5 MEM→WB MEM (Store) MEM→MEM

Load-Use 冒险

Load 指令的结果在 MEM 级末尾才可用，而后续指令在 EX 级开始就需要这个值——这是最坏情况的数据冒险：

1
2
3
4

LW   x1, 0(x2)     // IF ID EX MEM WB
ADD  x3, x1, x4    //    IF ID EX MEM WB
                         ↑ x1 在 MEM 级才可用
                         ↑ EX 级需要 x1

解决方案：

  1. 转发 + 插入一个气泡（最常用）
  2. 编译器调度：在 Load 和后一条之间插入不相关的指令
1
2
3

LW   x1, 0(x2)     // IF ID EX MEM WB
NOP                //    IF ID ST ST EX MEM WB (bubble)
ADD  x3, x1, x4    //       IF ID ── EX MEM WB

编译器可以通过调度隐藏这个延迟：

1
2
3

LW   x1, 0(x2)     // 先 Load
LW   x5, 0(x6)     // 插入另一个不相关的 Load
ADD  x3, x1, x4    // 这时 x1 已经就绪
🧓
Loader-Use 冒险是你做 RV32I 核心时必然会遇到的真问题。转发逻辑是 EX 级 datapath 中最容易出错的部分。做 RTL 时建议画清楚转发 MUX 的接线图和检测条件，把 5 种转发情况列全了再写代码——少一种就会出很难 debug 的 bug。

C.3 控制冒险

分支指令的处理

分支指令在 EX 级确定条件是否满足、在 MEM 级确定目标地址：

1
2

BEQ x1, x2, TARGET    // IF ID EX MEM WB
                      //    IF ID EX MEM WB (取 TARGET 地址)

每次分支都插入 2 个气泡 → 性能损失极大。

分支预测基础

预测策略 误预测惩罚 平均 CPI 影响
无预测（总是 stall） 2 cycles 0.4–0.7 CPI 增加
预测跳转 2 cycles (预测错时) 0.3–0.5 CPI 增加
预测不跳转 2 cycles (预测错时) 0.3–0.5 CPI 增加
1-bit 预测 2 cycles 0.1–0.2 CPI 增加
2-bit 预测 2 cycles 0.05–0.1 CPI 增加

减少分支惩罚的硬件技术

提前分支条件计算：将分支的条件比较从 EX 提前到 ID 级完成（硬件上加一个分支条件计算单元）。

分支目标地址计算提前到 ID：将 PC + offset 的计算从 EX 提前到 ID 级。

结合这两项改进 → 分支误预测惩罚从 2 cycles 减少到 1 cycle。

延迟分支 (Delayed Branch)

编译器在分支指令后面放置一条总是在分支前执行的指令：

1
2
3
4
5

// 延迟分支（槽中的指令总是执行）
BEQ x1, x2, TARGET
ADD x3, x4, x5    // 分支延迟槽 — 总是执行
TARGET:
SUB x6, x7, x8
  • 分支延迟槽的数量 = 分支确认前的流水级数
  • 编译器需要从分支前的代码中找一条不相关的指令放到延迟槽里
  • RISC-V 没有延迟分支（简化设计），但 MIPS 有
🧓
我们的 RV32I 核心你要决定要不要延迟分支。RISC-V 官方是「无延迟槽」的简化设计，但如果你追求高性能，加上延迟分支可以省一些转发逻辑。作为学习项目，建议先按 RISC-V 原版走，不要延迟槽。

C.4 流水线控制：暂停与冲刷

暂停 (Stall / Bubble)

在流水线中插入一个空操作，保持流水线数据不丢失。

硬件实现：

  • 在 ID/EX 寄存器中设置控制信号为 0（类似 NOP）
  • 保持 PC 不变（IF 级重复执行同一取指）
  • 保持 IF/ID 寄存器不变（ID 级重复译码）

冲刷 (Flush)

清空流水线中某些级的指令（如分支预测错误）：

硬件实现：

  • 将 ID/EX、EX/MEM、MEM/WB 寄存器清零
  • 重置 PC 到正确目标地址
  • 从新的 PC 开始取指

控制冒险的流水线实现

分支指令通过流水线时：

  1. IF 级：取分支指令
  2. ID 级：译码并读取寄存器
  3. EX 级：比较条件 + 计算目标地址
  4. 如果分支不跳转 → 继续顺序执行
  5. 如果分支跳转 → 冲刷 IF 和 ID 级，从目标地址重新取指

C.5 异常处理

精确异常 (Precise Exception)

异常发生时，流水线中的指令状态必须与程序顺序执行一致：

  1. 异常指令之前的所有指令都已完成提交
  2. 异常指令本身被标记为异常
  3. 异常指令之后的所有指令都未提交

5 级流水线中的精确异常实现：

流水级 异常处理
IF 页错误 → 标记指令，进入 ID 后处理
ID 非法指令 → 标记，进入 EX 后处理
EX 算术异常 → 标记，进入 MEM 后处理
MEM 数据访问异常 → 标记，进入 WB 前处理
WB 异常确认 → 清空后续指令，保存异常 PC

异常优先级

同一周期多个流水级同时发生异常时，按程序顺序优先级：

Exception Priority=IF<ID<EX<MEM\text{Exception Priority} = IF < ID < EX < MEMException Priority=IF<ID<EX<MEM

即 MEM 级异常优先级最高（最「老」的指令）。

精确异常的实现困难

  1. 多个异常同时发生：需要确定哪个异常先报告
  2. 自修改代码：指令覆盖了后面待取的指令
  3. 中断 (Interrupt)：外部中断需要流水线边界清理后才响应

C.6 流水线的局限与改进

深流水线的代价

流水级数 频率收益 分支误预测惩罚 转发复杂度
5 1x 2 cycles 低
10 ~1.5x 7 cycles 中
15 ~2x 12 cycles 高
20+ (Pentium 4) ~2.5x 17+ cycles 极高

Pentium 4 的经验：20 级流水线 → 分支误预测惩罚 20+ cycles → 分支预测需要 95%+ 的准确率才能弥补。

提高流水线效率的途径

  1. 转发：消除大多数 RAW 冒险的停顿
  2. 分支预测：降低控制冒险的影响
  3. 预取 (Prefetch)：隐藏指令缓存的缺失延迟
  4. 多发射：每个周期执行多条指令
  5. 乱序执行：绕过流水线的阻塞点

C.7 综合：RV32I 核心的流水线设计

我们的 RV32I 核心采用什么流水线？

推荐选择：

  • 5 级流水线：IF → ID → EX → MEM → WB
  • 全转发：5 种转发路径全部实现
  • 简单分支预测：2-bit 饱和计数器
  • 无延迟槽：RISC-V 标准
  • 精确异常：支持

流水线模块划分

模块 功能 关键信号
PC 程序计数器 PC, PC_next, Stall_PC
IF 指令存储器 IMEM_Addr, IMEM_Data
IF/ID 流水线寄存器 IR, PC+4
ID 译码 + 寄存器文件 A, B, Imm, func
ID/EX 流水线寄存器 A, B, Imm, func
EX ALU + 转发 ALU_Result, Zero
EX/MEM 流水线寄存器 ALU_Result, B (for Store)
MEM 数据存储器 DMEM_Addr, DMEM_Data
MEM/WB 流水线寄存器 ALU_Result, DMEM_Data
WB 写回 Reg_Write_Data

转发与控制信号

信号 作用 来源
ForwardA ALU 输入 A 的转发选择 转发检测单元
ForwardB ALU 输入 B 的转发选择 转发检测单元
Stall_F IF 级暂停信号 冒险检测单元
Stall_D ID 级暂停信号 冒险检测单元
Flush_E EX 级冲刷信号 分支预测结果

C.8 常见谬误与陷阱

  1. 认为更多流水级总是更好：深度与分支惩罚的权衡，过深适得其反
  2. 忽略 Load-Use 冒险：编译器调度可以隐藏但硬件也必须支持
  3. 低估转发逻辑的复杂度：5 级流水线的转发 MUX 还简单，更深的需要交叉开关
  4. 认为分支预测不如编译器调度重要：动态预测对未知代码模式至关重要
  5. 忽略异常和中断的精确性要求：设计验证时必须覆盖指令边界处的异常
  6. 误认为所有流水线停顿都是等价的：结构冒险、数据冒险和控制冒险的恢复机制不同
  7. 低估自修改代码的影响：指令缓存一致性问题在 RISC 设计中被简化，但在某些场景仍需处理

C.9 总结

本附录核心要点：

  1. 5 级流水线是处理器设计的入门基础：IF→ID→EX→MEM→WB
  2. 三类冒险（结构、数据、控制）是阻碍流水线连续运行的根本原因
  3. 转发消除大多数数据冒险（除了 Load-Use）
  4. 分支预测减少控制冒险的惩罚
  5. 精确异常保证异常时流水线状态与程序顺序一致
  6. 我们的 RV32I 核心应该实现：5 级流水线 + 全转发 + 2-bit 分支预测 + 无延迟槽 + 精确异常
