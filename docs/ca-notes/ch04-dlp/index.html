计算机体系结构学习笔记 — 第4章 数据级并行

计算机体系结构学习笔记 — 第4章 数据级并行

原著: Computer Architecture: A Quantitative Approach, 6th Edition — John L. Hennessy & David A. Patterson

数据级并行（DLP）利用同一条指令同时操作多个数据，是 SIMD、向量处理器和 GPU 的核心技术。

4.1 引言 — 什么是数据级并行

DLP 与 ILP 的区别

特性 ILP (指令级并行) DLP (数据级并行)
并行单位 多条不同指令 一条指令操作多个数据
硬件复杂度 高（乱序、调度） 低（控制简单）
适用场景 控制流丰富的代码 数据密集的计算
功耗效率 低（每单位性能能耗高） 高
典型应用 操作系统、数据库 图形、ML、信号处理

DLP 的核心思想：如果对大量数据做相同的操作，与其花巨大的代价用 ILP 去挖掘并行度，不如设计专用的 SIMD 数据路径，一条指令同时操作 N 个数据元素。

🧓
这就是你做 RV32I 核心时应该记住的：对于大多数控制流密集的应用（OS、调度），ILP 很关键；但对于 ML 推理、矩阵运算这些数据密集的任务，DLP 才是效率更高的方案。 你在华为/英特尔做过的芯片里基本都有 SIMD 指令——ARM 的 NEON、x86 的 SSE/AVX。这些都是 DLP 的具体实现。

三种 DLP 架构

架构 典型代表 数据并行方式
向量架构 Cray-1, NEC SX 向量寄存器 + 向量指令
SIMD 扩展 x86 SSE/AVX, ARM NEON 短向量固定宽度 SIMD
GPU NVIDIA CUDA, AMD SIMT (Single Instruction, Multiple Threads)

4.2 向量处理器 (Vector Processors)

向量架构的基本思想

向量处理器拥有向量寄存器——每个向量寄存器可存储多个标量元素（如 64 个 64-bit 浮点数）。

1
2
3
4
5
6
7
8
9

// 标量版本
for (i = 0; i < 64; i++)
    C[i] = A[i] + B[i];

// 向量版本（伪汇编）
VLOAD v1, A      // 加载 A[0..63] 到向量寄存器 v1
VLOAD v2, B      // 加载 B[0..63] 到向量寄存器 v2
VADD  v3, v1, v2 // v3 = v1 + v2 (64 个加法同时执行)
VSTORE v3, C     // 写回 C[0..63]

向量架构的关键特性

向量长度寄存器 (VL)：支持可变长度的向量操作，处理不规则的循环剩余部分。

向量屏蔽寄存器 (VM)：支持条件执行（选择性操作）。

1
2
3

// 条件向量操作：只对 A[i] > 0 的元素做加法
VNESGT VM, v1, #0  // 比较 A[i] > 0，结果存入 VM
VADD  v3, v1, v2   // 只在 VM 为真的位置上执行加法

向量链接 (Chaining)：一条向量指令的结果直接作为下一条的输入，无需等待全部完成。这是向量处理器中的「转发」机制。

向量处理器的优势

优势 说明
控制简单 一条向量指令替代整个循环，减少了取指和译码开销
结构规整 访存模式可预测，支持高效预取和流式访问
延迟容忍 能通过并行处理多个向量操作来隐藏内存延迟
功耗效率 控制逻辑占比低，计算密度高

向量处理器的局限

  • 不适合不规则数据访问（稀疏矩阵、链表遍历）
  • 不适合控制流密集的代码
  • 向量寄存器文件面积大

4.3 SIMD 指令集扩展

x86 SIMD 演进

扩展 宽度 寄存器数 支持类型 发布
MMX 64-bit 8 整数 1997
SSE 128-bit 8 (XMM) 单精度浮点 1999
SSE2 128-bit 8 双精度浮点 + 整数 2001
AVX 256-bit 16 (YMM) 浮点 + 整数 2011
AVX-512 512-bit 32 (ZMM) 浮点 + 整数 + 掩码 2017

ARM NEON

  • 128-bit SIMD 引擎（可视为 2×64-bit 或 4×32-bit）
  • 32 个 128-bit 寄存器（AArch64）
  • 支持 8/16/32/64-bit 整数和单精度/双精度浮点
  • 广泛应用在移动 SoC（你的华为/海思芯片里的麒麟系列就用了 NEON）

RISC-V 向量扩展 (RVV)

RISC-V V 扩展（“V” 表示向量）采用可伸缩向量架构：

  • 向量长度在硬件和软件之间解耦
  • 软件只需设置 VLEN（向量长度），硬件决定如何映射到具体实现
  • 标量处理器可以在 128-bit 实现，服务器处理器可以在 4096-bit 实现
  • 同一套二进制代码在两种实现上都能跑
1
2
3
4
5
6

// RISC-V V 扩展向量加法
vsetvli t0, a2, e32  // 设置向量长度 (VL), 32-bit 元素
vle32.v v1, (a0)     // 加载向量
vle32.v v2, (a1)     // 加载向量
vadd.vv v3, v1, v2   // 向量加法
vse32.v v3, (a3)     // 存储结果
🧓
RISC-V V 扩展的设计很优雅——它不像 x86 那样固定宽度（AVX-256 → AVX-512 不兼容），而是让软件写"我这次需要处理 8 个 32-bit 整数"，硬件自行决定怎么实现。 如果你以后给 RV32I 核心加上向量扩展，V 扩展是默认选择。

SIMD 编程挑战

自动向量化：编译器自动识别可向量化的循环。

限制自动向量化的因素：

  • 循环携带依赖（本次迭代 = 下次迭代）
  • 指针别名（编译器不确定是否重叠）
  • 复杂控制流（内部有多个条件分支）
  • 不规则数据访问（间接寻址）
1
2
3
4
5
6
7

// 可向量化
for (i = 0; i < N; i++) 
    a[i] = b[i] + c[i];

// 不可向量化（循环携带依赖）
for (i = 1; i < N; i++) 
    a[i] = a[i-1] + b[i];

4.4 GPU 架构

GPU 的 DLP 方式：SIMT

SIMT (Single Instruction, Multiple Threads)：

  • 一个指令被一组线程（warp/wavefront）同时执行
  • 每个线程有自己的程序计数器（逻辑上）和一套寄存器
  • 条件分支通过分支发散和分支汇聚处理

GPU 架构层次 

           GPU (设备)
    ┌─────────────────────┐
    │    Stream Proc      │
    │    Cluster (SPC)    │ ← 多个
    │  ┌───────────────┐  │
    │  │ Shader Proc   │  │
    │  │ Multiplex     │  │
    │  │ (SM/SIMD Core)│  │ ← 多个 SPC 内
    │  │  ┌───┐ ┌───┐  │  │
    │  │  │SP │ │SP │...│  │ ← 一组流处理器
    │  │  └───┘ └───┘  │  │
    │  └───────────────┘  │
    └─────────────────────┘

NVIDIA GPU 架构 (以 A100 为例)

参数 值
CUDA 核心数 6912
SM (Streaming Multiprocessor) 数 108
每个 SM 的 CUDA 核心数 64
Warp 大小 32 线程
寄存器/SM 65536
共享内存/SM 192 KB
L2 缓存 40 MB
HBM2 显存 80 GB
显存带宽 2 TB/s

GPU 性能瓶颈

计算瓶颈：

  • 浮点吞吐（FLOPS）受限于核心数和频率
  • 整数和特殊函数单元可能成为瓶颈

内存瓶颈：

  • 全局内存带宽（HBM 提供数 TB/s 带宽但仍可能不足）
  • 共享内存大小有限
  • 寄存器压力

控制流瓶颈：

  • Warp 发散：同一 warp 内不同线程走不同路径，所有路径串行执行
1
2
3
4
5
6
7

// warp 发散示例
if (threadIdx.x % 2 == 0) {
    // 一半线程进入
} else {
    // 另一半线程进入
    // 两条路径串行化！性能损失 ~50%
}

GPU vs CPU 对比

特性 CPU GPU
设计目标 低延迟单线程 高吞吐并行
控制逻辑 复杂（乱序、分支预测） 简单（SIMT）
缓存 大（减少延迟） 小（辅助带宽）
ALU 占比 小（~10%） 大（~80%）
内存带宽 中等 极高
典型 TDP 65–150W 75–450W
典型应用 通用计算 图形、ML、HPC

矩阵乘法

对于矩阵乘法，GPU 的理论峰值 FLOPS 通常比 CPU 高 5–10 倍。

CNN 卷积

卷积层是 DLP 的自然应用：

  • 相同卷积核对整张特征图的所有像素做操作
  • 批量输入可以进一步并行化
  • GPU 在 CNN 推理和训练中占主导

图像与信号处理

  • 每个像素独立处理 → 完美 DLP
  • 色彩空间转换、滤波、变换

4.6 综合案例：向量化与自动向量化

向量化效率

向量化效率 = 实际执行数据量 / 理论最大执行数据量

影响效率的因素：

  • 剩余循环：当 N 不能被向量宽度整除
  • 数据对齐：非对齐访问可能触发分段加载
  • 缩减操作 (Reduction)：求和、求最大值等需要多个步骤

性能收益实例

操作 标量 128-bit SIMD 256-bit SIMD 512-bit SIMD
单精度加法 1 4 8 16
双精度乘法 1 2 4 8
整数加法 1 4–16 8–32 16–64

实际加速比受限于：

  • 向量化比例（Amdahl 定律）
  • 数据搬运开销
  • 循环开销节省

编译器向量化报告

GCC 和 LLVM 都可以报告哪些循环被向量化了：

1

gcc -O2 -fopt-info-vec-missed -fopt-info-vec program.c

4.7 DLP 的功耗效率

数据并行架构的功耗效率远优于传统超标量：

架构 每瓦性能 (GFLOPS/W, ~7nm)
CPU (超标量) 5–20
CPU (向量化) 20–50
GPU 50–200
张量处理单元 (TPU) 200–1000+

原因：DLP 架构用更多晶体管做计算（ALU），更少做控制逻辑。

🧓
这也是华为做鲲鹏和昇腾的底层逻辑——通用 CPU 的每瓦性能增长已经到头了，用 DSA（领域专用架构）做数据密集型计算才能继续提升每瓦性能。 你做综合时关注的功耗优化（DVFS、时钟门控），在 DLP 架构下效果更显著，因为计算单元占比高、控制单元占比低。

4.8 常见谬误与陷阱

  1. 认为 SIMD 宽度加倍性能就能翻倍：Amdahl 定律限制 + 剩余循环 + 带宽瓶颈
  2. 假设所有循环都能向量化：指针别名、依赖、控制流都可能是障碍
  3. 把 GPU 的峰值 FLOPS 当实际性能：内存带宽、数据搬运、kernel 启动开销都影响实际吞吐
  4. 忽略数据搬运开销：CPU ↔ GPU 的数据传输是许多应用的瓶颈
  5. 认为 GPU 适合所有并行任务：细粒度控制流密集的任务在 GPU 上表现很差
  6. 低估 warp 发散的影响：条件分支多的代码在 GPU 上效率极低
  7. 过度依赖自动向量化：编译器自动向量化能力有限，手工 intrinsic 写 SIMD 代码可能效果更好

4.9 总结

本章核心要点：

  1. DLP 与 ILP 是正交的并行维度——ILP 挖掘指令间的并行，DLP 利用同指令多数据
  2. 三种 DLP 架构：向量处理器（长向量）、SIMD 扩展（短向量固定宽度）、GPU（SIMT）
  3. 向量处理器通过向量寄存器、链接、屏蔽寄存器提供可扩展的高效数据并行
  4. SIMD 扩展（SSE/AVX/NEON/RVV）是通用处理器上的主流 DLP 方案
  5. GPU 通过大规模 SIMT 和分级存储（全局/共享/私有）实现极高的并行吞吐
  6. 向量化效率受限于数据类型、对齐、剩余循环和编译器能力
  7. DLP 的功耗效率远优于 ILP——用更多晶体管做计算、更少做控制
